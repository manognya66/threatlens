Broken Access Control - Insecure Direct Object Reference (IDOR)

Vulnerability Summary

The application exposes a Broken Access Control vulnerability in the basket API endpoint. An authenticated user can modify the basket identifier in the request URL to access other users' shopping carts.

The application does not verify ownership of the requested basket resource, allowing unauthorized access to sensitive data belonging to other users. This is a classic Insecure Direct Object Reference (IDOR).

Target: OWASP Juice Shop  |  Tool: Burp Suite  |  Environment: Kali Linux  |  OWASP Category: A01 – Broken Access Control

Attack Scenario

A malicious authenticated user discovers that the basket endpoint uses a sequential integer ID in the URL. By incrementing or decrementing this ID, they can access baskets belonging to other users without any authorisation error.

1

Attacker's own basket request

After adding an item to their cart, the attacker's authenticated request goes to:

HTTP Request
GET /rest/basket/6 HTTP/1.1
Host: localhost:3000
Authorization: Bearer <JWT_TOKEN>
2

Manipulated request

The attacker changes /6 to /1 to target another user's basket:

HTTP Request — Modified
GET /rest/basket/1 HTTP/1.1
Host: localhost:3000
Authorization: Bearer <JWT_TOKEN>
3

Server returns another user's data

No ownership check is performed — the server returns basket data belonging to UserId: 1 to a different authenticated user.

Steps to Reproduce

Step 1 — Register a New User Account

Test Credentials
email:    user2@gmail.com
password: User2

Step 2 — Log In and Add a Product to Basket

Log in with the registered account, navigate to the products page and click Add to Basket. This triggers a request that includes the basket identifier.

Step 3 — Intercept the Request in Burp Suite

With Burp Suite Proxy active, capture the request triggered by the basket action:

Captured Request 
GET /rest/basket/6 HTTP/1.1
Host: localhost:3000
Authorization: Bearer <JWT_TOKEN>

The basket identifier (6) is exposed directly in the URL — no obfuscation or token-based reference.

Step 4 — Send to Repeater and Modify the ID

Right-click the request → Send to Repeater. Change the basket ID from 6 to 1 and send.

Burp Repeater lets you manually resend and edit HTTP requests without going through the browser — essential for testing parameter manipulation.

Proof of Exploitation

The server returns HTTP 200 OK with basket data belonging to a different user:

Server Response 
{
  "status": "success",
  "data": {
    "id": 1,
    "UserId": 1,
    "Products": [
      {
        "id": 1,
        "name": "Apple Juice (1000ml)",
        "price": 1.99
      }
    ]
  }
}
The response shows UserId: 1 — but the authenticated user making the request is user2. The server returned another user's basket without any authorization check.

Impact

  • Unauthorized access to other users' purchase data
  • Exposure of user activity, items, and purchasing patterns
  • Privacy violations and potential data leakage
  • In production: exposure of customer orders and personal information

Root Cause

The application fails to enforce authorization checks when retrieving basket resources. The server trusts the user-provided basket identifier instead of verifying ownership.

Vulnerable Logic

Python — Vulnerable 
# ❌ No ownership check — anyone can access any basket ID
if basketId exists:
    return basket

Secure Logic

Python — Secure 
# ✅ Ownership verified before returning data
if basket.owner == logged_in_user:
    return basket
else:
    return 403 Forbidden

Mitigation

Recommended Fixes

  • Verify resource ownership — compare the basket's userId against the authenticated user's ID before returning any data
  • Avoid sequential IDs — use UUIDs or non-guessable indirect references instead of incrementing integers
  • Implement access control at API level — authorization checks must happen server-side, not just in the frontend
  • Use indirect object references — map user-visible IDs to internal IDs server-side
  • Return 403 Forbidden — never return 200 with empty data for unauthorized requests; always deny explicitly
Secure fix example:
if basket.userId == authenticatedUserId: return basket else: return 403
