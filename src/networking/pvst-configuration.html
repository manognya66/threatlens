What I Learned

I configured PVST across three switches in a redundant triangle topology. The key insight: instead of one spanning tree for all VLANs (like CST), PVST creates a separate spanning tree for each VLAN — allowing different root bridges and enabling load balancing.

Watching each VLAN elect a different root bridge was fascinating. VLAN 10 used Switch A as root, VLAN 20 used Switch B, and VLAN 30 used Switch C. Traffic distribution across the physical topology improved dramatically.

What is PVST?

PVST (Per-VLAN Spanning Tree) is a Cisco proprietary protocol that runs a separate instance of Spanning Tree Protocol for each VLAN. Unlike CST (Common Spanning Tree), which creates one spanning tree for all VLANs, PVST allows each VLAN to have its own root bridge and topology.

Why Do We Need PVST?

The Problem: Without spanning tree, a redundant switched network with VLANs will create Layer 2 loops. Broadcasts will circulate forever, causing broadcast storms and MAC table instability. VLAN 20 alone can bring down the entire network.

CST vs PVST

Feature CST (802.1D) PVST
Spanning Tree Instances 1 (for all VLANs) 1 per VLAN
Root Bridge Single root for all VLANs Different root per VLAN
Load Balancing No Yes (across VLANs)
Link Utilization Some links always blocked Different links blocked per VLAN
CPU/Memory Usage Lower Higher (one instance per VLAN)
Vendor Support All vendors Cisco proprietary

PVST Load Balancing Example

In my lab with 3 VLANs:

  • VLAN 10: Switch A is root → certain paths are active
  • VLAN 20: Switch B is root → different paths are active
  • VLAN 30: Switch C is root → yet different paths are active

This distributes traffic across all links instead of blocking the same links for all VLANs.

PVST Key Concepts

Term What It Means
Root Bridge Switch with lowest bridge ID (elected per VLAN in PVST)
Bridge Priority Value 0-61440 in increments of 4096 (lower is better)
Hello Time How often BPDUs are sent (default: 2 seconds)
Max Age How long to wait before declaring a BPDU stale (default: 20 sec)
Forward Delay Time spent in Listening and Learning states (default: 15 sec each)
BPDU Bridge Protocol Data Unit — messages switches use to elect root

Spanning Tree Port States

State Forwards Traffic Learns MAC Duration
Disabled No No —
Blocking No No 20 sec (Max Age)
Listening No No 15 sec (Forward Delay)
Learning No Yes 15 sec (Forward Delay)
Forwarding Yes Yes —

My Lab Topology

Three switches in a triangle (full mesh) with 9 computers spread across 3 VLANs. Each switch connects to all other switches, creating redundancy and potential loops.

end

Switch B — Root for VLAN 20 
! Enable PVST mode

! Set as root bridge for VLAN 20

! Save configuration
end
write memory
Switch C — Root for VLAN 30 
! Enable PVST mode
spanning-tree mode pvst

! Set as root bridge for VLAN 30
spanning-tree vlan 30 priority 4096

! Save configuration
end
write memory
Priority Values: Bridge priority must be a multiple of 4096. Valid values: 0, 4096, 8192, 12288, ..., 61440. Lower priority = higher preference to become root bridge.

Step 4 — Tune PVST Timers (Optional)

Tuning Timers — All Switches 
! Adjust hello time for VLAN 10 (faster convergence)
spanning-tree vlan 10 hello-time 1
! Default is 2 seconds

! Adjust max-age for VLAN 20
spanning-tree vlan 20 max-age 6
! Default is 20 seconds

! Adjust forward-time for VLAN 30
spanning-tree vlan 30 forward-time 4
! Default is 15 seconds

end
write memory
Warning: Lowering timers too much can cause instability. The formula is: Max Age ≥ 2 × (Hello Time + 1). Forward Delay should be at least 4 seconds.

After configuration, I ran show spanning-tree vlan 10 on Switch A and confirmed it was the root bridge. Switch B and C showed their ports in different states (some forwarding, some blocking) for each VLAN, proving PVST was working correctly.

Key Takeaways

What I Understood

  • PVST runs a separate spanning tree instance for each VLAN
  • Different VLANs can have different root bridges for load balancing
  • Priority must be a multiple of 4096 (0, 4096, 8192, etc.)
  • Lower priority value = higher chance of becoming root bridge
  • All trunk ports must have dot1q encapsulation configured
  • Tuning timers can speed up convergence but risks instability
  • Without spanning tree, loops will crash the network in seconds

How PVST Achieves Load Balancing

In my lab:

  • VLAN 10 traffic: Flows through Switch A paths (A is root)
  • VLAN 20 traffic: Flows through Switch B paths (B is root)
  • VLAN 30 traffic: Flows through Switch C paths (C is root)

Instead of all traffic going through one root bridge, it's distributed across all three switches.

PVST vs RPVST+ vs MST

Protocol Convergence Scalability Use Case
PVST Slow (50 sec) Medium Small networks, Cisco only
RPVST+ Fast (1-2 sec) Medium Recommended for most Cisco networks
MST Fast High Large networks with many VLANs

Common Mistakes I Made

  • Forgot trunk encapsulation: Ports stayed in access mode until I added switchport trunk encapsulation dot1q
  • Used wrong priority value: Tried 5000 — Cisco rejected it because it's not a multiple of 4096
  • Timer confusion: Set hello-time too low relative to max-age, causing flapping
  • Didn't verify per VLAN: Checked only show spanning-tree instead of checking each VLAN separately
