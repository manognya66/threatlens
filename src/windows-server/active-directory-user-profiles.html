What I Learned

I set up Active Directory on Windows Server and created user accounts that could log into a domain-joined client machine. Before this, I never really understood why enterprises use domain networks instead of local accounts — now it's obvious.

Managing one user account from a central server and having it work on any machine in the network is genuinely powerful. This is the foundation of every enterprise IT environment.

What is Active Directory?

Active Directory (AD) is a directory service built into Windows Server. It stores information about users, computers, and resources in a network — and controls who can access what.

Think of AD as the central ID card system for an organization. Every user gets one account, and that account works on any computer joined to the domain.

Key Concepts

Term What It Means
Domain A logical group of network objects (users, computers) managed from one place
Domain Controller (DC) The server running Active Directory — authenticates all logins
OU (Organizational Unit) A folder inside AD to organize users, groups, or computers
User Profile The desktop, settings, and files attached to a user account
Domain Join Connecting a client PC to the domain so it can use AD accounts

Types of User Profiles

Profile Type Where It's Stored What It Does
Local Profile On the client PC Settings only available on that one machine
Roaming Profile On the server Settings follow the user to any machine in the domain
Mandatory Profile On the server (read-only) Same fixed settings for all users, changes don't save

My Lab Setup

I used two virtual machines on the same network — one running Windows Server 2019 as the Domain Controller, and one running Windows 10 as the client.

Machine OS Role IP Address
Server-1 Windows Server 2019 Domain Controller 192.168.1.1
Client-1 Windows 10 Domain Member 192.168.1.10
Domain name I used: threatlens.local
The client's DNS must point to the Domain Controller's IP, otherwise domain join will fail.

Configuration Steps

Step 1 — Install Active Directory Domain Services

On the Windows Server machine, I installed the AD DS role through Server Manager.

1

Open Server Manager

Click Start → Server Manager

2

Add Roles and Features

Click Manage → Add Roles and Features

3

Select AD DS

Choose Role-based installation → select your server → tick Active Directory Domain Services → Install

4

Promote to Domain Controller

After install, click the flag icon in Server Manager → Promote this server to a domain controller

5

Create a New Forest

Select Add a new forest → Enter domain name (e.g. threatlens.local) → Set DSRM password → Install → Server restarts

Step 2 — Create an Organizational Unit (OU)

I created an OU to keep my domain users organized separately from default containers.

1

Open ADUC

Go to Start → Windows Administrative Tools → Active Directory Users and Computers

2

Create New OU

Right-click your domain (threatlens.local) → New → Organizational Unit → Name it (e.g. ThreatLens Users)

Step 3 — Create User Accounts

Inside the OU, I created individual user profiles for domain login.

1

Right-click the OU

Right-click ThreatLens Users → New → User

2

Fill in User Details

Enter First name, Last name, and User logon name (e.g. jdoe@threatlens.local) → Next

3

Set Password

Enter a password → Choose options:

  • User must change password at next logon — good for real users
  • Password never expires — useful for lab/service accounts

4

Finish

Click Next → Finish. The user account now appears inside the OU.

Step 4 — Join the Client PC to the Domain

On the Windows 10 client, I pointed DNS to the Domain Controller and then joined the domain.

1

Set DNS to the Domain Controller

Go to Network Settings → Change adapter options → IPv4 Properties → Set Preferred DNS to 192.168.1.1

2

Open System Properties

Right-click This PC → Properties → Change settings → Change

3

Join the Domain

Select Domain → Type threatlens.local → Enter domain admin credentials when prompted → OK

4

Restart

Client will show "Welcome to the threatlens.local domain" → Restart to apply

Step 5 — Log In with a Domain User Account

After the client restarts, I logged in using the domain account I created.

Windows Login 
! At the Windows login screen, click "Other user"
! Then log in using:
Username: threatlens\jdoe
          (or) jdoe@threatlens.local
Password: [the password you set]
First login takes a moment — Windows is creating a local profile for the domain user on that machine. After that, the user's desktop and settings are stored.

Step 6 — Configure a Roaming Profile (Optional)

I also experimented with roaming profiles so the user's settings follow them across machines.

1

Create a Shared Folder on the Server

Create a folder (e.g. C:\Profiles) → Right-click → Properties → Sharing → Advanced Sharing → Share it → Set permissions so domain users can read/write

2

Set Profile Path in ADUC

In Active Directory Users and Computers → double-click the user → go to Profile tab → Set Profile path to \\Server-1\Profiles\%username%

3

Test It

Log in on Client-1 → Change wallpaper or create a file on Desktop → Log in on another machine → Same settings appear

Common mistake: Share permissions and NTFS permissions are different. I had to set both — Share permission to Everyone: Full Control and NTFS permission to the specific user or Domain Users group.

Verification Commands

I used these commands to verify the setup was working correctly.

Command Prompt — Client PC 
! Check if the PC is joined to the domain
systeminfo | findstr /B /C:"Domain"

! Check currently logged-in domain user
whoami

! Verify DNS is resolving the domain controller
nslookup threatlens.local

! Ping the Domain Controller
ping 192.168.1.1
PowerShell — Server 
# List all users in Active Directory
Get-ADUser -Filter * | Select Name, SamAccountName

# Check a specific user
Get-ADUser -Identity jdoe

# See which computers are joined to the domain
Get-ADComputer -Filter * | Select Name

Key Takeaways

What I Understood

  • The client's DNS must point to the Domain Controller — domain join fails without this
  • OUs help organize users but don't affect permissions on their own
  • Roaming profiles need both Share and NTFS permissions set correctly
  • Domain accounts work on any domain-joined machine — local accounts don't
  • whoami and systeminfo are the fastest ways to verify domain membership
