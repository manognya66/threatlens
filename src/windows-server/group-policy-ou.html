What I Learned

After setting up Active Directory and creating user accounts, I wanted to understand how administrators enforce rules across machines without logging into each one individually. Group Policy is exactly that.

I created a GPO, linked it to an OU, and watched it push settings to the client machine automatically. What took me a while to understand was that the policy doesn't apply instantly — you either wait for the refresh cycle or force it manually with gpupdate.

What is Group Policy?

A Group Policy Object (GPO) is a set of rules defined on the Domain Controller that automatically applies settings to users or computers in a domain. It saves you from manually configuring every machine.

Think of a GPO as a rulebook. You write the rules once on the server, link it to an OU, and every user or computer inside that OU automatically follows those rules.

Key Concepts

Term What It Means
GPO Group Policy Object — a collection of policy settings
GPMC Group Policy Management Console — the tool used to create and link GPOs
GPME Group Policy Management Editor — where you configure the actual settings inside a GPO
Link Connecting a GPO to a Site, Domain, or OU so it takes effect
gpupdate Command to force immediate refresh of Group Policy on a client
gpresult Command to see which policies are currently applied to a user or computer

Where GPOs Can Be Linked

Scope Affects Example Use
Site All objects in a physical location Apply proxy settings to a whole office
Domain All users and computers in the domain Enforce password policy for everyone
OU Only users/computers in that specific OU Block Control Panel for a specific department

GPO Processing Order — LSDOU

GPOs are applied in this order. Later policies override earlier ones if they conflict:

L

Local

Settings on the local machine itself

S

Site

GPOs linked to the AD site

D

Domain

GPOs linked to the domain

OU

Organizational Unit

GPOs linked to the OU — these win if there's a conflict

My Lab Setup

I used the same two-machine setup from the Active Directory lab. The OU ThreatLens Users was already created with user jdoe inside it.

Machine OS Role IP Address
Server-1 Windows Server 2019 Domain Controller 192.168.1.1
Client-1 Windows 10 Domain Member 192.168.1.10
What I was trying to achieve:
  • Create a GPO and link it to the ThreatLens Users OU
  • Use the GPO to disable Control Panel access for users in that OU
  • Set a custom desktop wallpaper for all users in the OU
  • Verify it applies on the client machine

Configuration Steps

Step 1 — Open Group Policy Management Console

1

Open GPMC on the Server

Go to Start → Windows Administrative Tools → Group Policy Management

2

Navigate to Your OU

Expand Forest → Domains → threatlens.local → ThreatLens Users

Step 2 — Create and Link a New GPO

1

Create the GPO

Right-click ThreatLens Users → Create a GPO in this domain, and Link it here

2

Name It

Give it a clear name like ThreatLens-UserPolicy → Click OK

3

Edit the GPO

Right-click the new GPO → Edit — this opens the Group Policy Management Editor

Step 3 — Disable Control Panel Access

Inside the Group Policy Management Editor, I navigated to the setting that blocks Control Panel.

1

Navigate to the Setting

User Configuration → Policies → Administrative Templates → Control Panel

2

Enable the Policy

Double-click "Prohibit access to Control Panel and PC settings" → Set to Enabled → Click OK

This setting lives under User Configuration, not Computer Configuration. That means it applies to the user account wherever they log in — not to a specific machine.

Step 4 — Set a Desktop Wallpaper via GPO

I also pushed a custom wallpaper to all users in the OU using a shared network path.

1

Store the Wallpaper on the Server

Save an image to a shared folder accessible by domain users — e.g. \\Server-1\Shared\wallpaper.jpg

2

Navigate to the Setting

User Configuration → Policies → Administrative Templates → Desktop → Desktop

3

Configure Wallpaper Policy

Double-click "Desktop Wallpaper" → Set to Enabled → Enter the wallpaper path: \\Server-1\Shared\wallpaper.jpg → Set wallpaper style to Fill → Click OK

Step 5 — Set a Password Policy for the Domain

Password policies are best applied at the Domain level, not the OU. I configured this separately through the Default Domain Policy.

1

Edit Default Domain Policy

In GPMC, right-click Default Domain Policy under the domain → Edit

2

Navigate to Password Settings

Computer Configuration → Policies → Windows Settings → Security Settings → Account Policies → Password Policy

3

Configure the Rules

Settings I changed:

  • Minimum password length: 8 characters
  • Password must meet complexity requirements: Enabled
  • Maximum password age: 30 days

Step 6 — Force Apply on the Client

By default, Group Policy refreshes every 90 minutes. I forced it immediately using gpupdate on the client.

Command Prompt — Client PC (run as administrator) 
! Force immediate Group Policy refresh
gpupdate /force

! Force refresh and log off to apply User Configuration policies
gpupdate /force /logoff
After running gpupdate /force and logging back in as jdoe, the wallpaper had changed and trying to open Control Panel showed an error message — the policy was working.

Verification Commands

These commands helped me confirm exactly which policies were applied and where they came from.

Command Prompt — Client PC 
! Show applied policies for the current user
gpresult /r

! Generate a detailed HTML report of applied GPOs
gpresult /h C:\gpreport.html

! Open the report in a browser
start C:\gpreport.html
PowerShell — Server 
# List all GPOs in the domain
Get-GPO -All

# See which GPOs are linked to a specific OU
Get-GPInheritance -Target "OU=ThreatLens Users,DC=threatlens,DC=local"

# Force GPUpdate on a remote client
Invoke-GPUpdate -Computer "Client-1" -Force
Tip: The gpresult /h HTML report is the clearest way to see which GPOs applied, which were blocked, and why. I found it much easier to read than the console output.

Key Takeaways

What I Understood

  • GPOs are applied in order: Local → Site → Domain → OU — OU wins conflicts
  • User Configuration follows the user account; Computer Configuration follows the machine
  • GPO changes don't apply instantly — use gpupdate /force to push immediately
  • Password policies work at the Domain level, not the OU level
  • gpresult /h is the best way to troubleshoot which policies are actually active
  • Always name GPOs clearly — it becomes confusing fast when you have many
