What I Learned

I set up a web server (IIS) and an FTP server on Windows Server in a domain network. The biggest realization: IIS isn't just for hosting websites — it can also serve FTP sites, and both can authenticate users from Active Directory.

Configuring FTP with domain authentication took some trial and error, especially figuring out the difference between Basic Authentication (passwords sent in plaintext) and Windows Authentication (domain credentials).

What is IIS & FTP?

IIS (Internet Information Services)

IIS is Microsoft's web server software. It hosts websites, web applications, and APIs on Windows Server. You can run static HTML sites, ASP.NET apps, or even PHP applications.

FTP (File Transfer Protocol)

FTP is a protocol for transferring files between computers over a network. An FTP server lets users upload and download files remotely — useful for sharing documents, backups, or deploying website files.

IIS can host both HTTP (web) and FTP sites on the same server. In a domain network, both can use Active Directory for user authentication.

Key Concepts

Term What It Means
Default Website The default site created when IIS is installed (runs on port 80)
Application Pool Isolated process that runs web applications (crashes don't affect other sites)
FTP Site A folder accessible via FTP protocol (usually on port 21)
Virtual Directory A folder from another location that appears inside a website
Bindings IP address, port, and hostname a site listens on
Basic Auth Username/password sent in plaintext (not secure without SSL)
Windows Auth Uses domain credentials (encrypted, more secure)

Authentication Types

Type How It Works Security
Anonymous Anyone can access without login No security
Basic Username/password in plaintext Low (use with SSL only)
Windows Domain credentials (NTLM/Kerberos) High (encrypted)

My Lab Setup

Same two-machine lab from the Active Directory setup. The Domain Controller now also runs IIS and FTP.

Machine OS Role IP Address
Server-1 Windows Server 2019 DC + IIS + FTP 192.168.1.1
Client-1 Windows 10 Domain Member 192.168.1.10
What I was trying to achieve:
  • Install IIS and host a simple HTML website
  • Install FTP server on the same machine
  • Allow domain users to authenticate and upload files via FTP
  • Test access from the client machine

Configuration Steps

Part 1 — Installing IIS (Web Server)

1

Open Server Manager

Click Start → Server Manager

2

Add IIS Role

Manage → Add Roles and Features → Role-based installation → Select your server → Check Web Server (IIS) → Add Features → Next → Install

3

Verify Installation

Open a browser on the server → Navigate to http://localhost → You should see the IIS default welcome page

Part 2 — Creating a Simple Website

1

Create a Folder for the Website

Create C:\inetpub\mywebsite → Inside, create a file called index.html

2

Add Simple HTML Content

Open index.html in Notepad and paste:

HTML 
<!DOCTYPE html>
<html>
<head>
    <title>ThreatLens Test Site</title>
</head>
<body>
    <h1>Welcome to ThreatLens!</h1>
    <p>This website is hosted on IIS in a domain network.</p>
</body>
</html>
3

Open IIS Manager

Start → Administrative Tools → Internet Information Services (IIS) Manager

4

Stop Default Website

Expand the server node → Expand Sites → Right-click Default Web Site → Stop

5

Create a New Site

Right-click Sites → Add Website

  • Site name: ThreatLens
  • Physical path: C:\inetpub\mywebsite
  • Port: 80

Click OK

6

Test the Website

From the client machine, open a browser → Navigate to http://192.168.1.1 → You should see your HTML page

When I browsed to the server's IP from the client, my custom HTML page loaded perfectly. IIS is now serving a website accessible across the domain network.

Part 3 — Installing FTP Server

1

Add FTP Feature to IIS

Server Manager → Add Roles and Features → Web Server (IIS) → Click Next until you reach Role Services → Expand FTP Server → Check FTP Service and FTP Extensibility → Install

2

Create FTP Root Folder

Create a folder: C:\FTPRoot → Inside, create a subfolder: Uploads

3

Set NTFS Permissions

Right-click C:\FTPRoot → Properties → Security → Edit → Add Domain Users → Grant Modify permissions → OK

Part 4 — Creating an FTP Site

1

Open IIS Manager

Expand the server node → Right-click Sites → Add FTP Site

2

FTP Site Information

FTP site name: ThreatLens FTP
Physical path: C:\FTPRoot
Click Next

3

Binding and SSL Settings

IP Address: All Unassigned
Port: 21
SSL: No SSL
Click Next

4

Authentication and Authorization

Authentication: Check Basic
Allow access to: Specified users
User: Domain Users (or specific user like threatlens\jdoe)
Permissions: Check both Read and Write
Click Finish

Important: Basic Authentication sends passwords in plaintext. For production, use SSL (FTPS) or switch to Windows Authentication. I used Basic here for simplicity in the lab.

Part 5 — Configure Windows Firewall for FTP

1

Open Firewall Settings

Control Panel → Windows Defender Firewall → Advanced Settings

2

Allow FTP Traffic

Click Inbound Rules → New Rule → Port → TCP → Specific local ports: 21 → Allow the connection → Apply to all profiles → Name it FTP Server → Finish

Part 6 — Testing FTP from the Client

1

Test with Windows Explorer

On Client-1, open File Explorer → In the address bar, type: ftp://192.168.1.1 → Press Enter

2

Log In

Enter domain credentials:
Username: threatlens\jdoe
Password: [the user's password]

3

Upload a Test File

Create a text file on your desktop → Drag it into the FTP folder in File Explorer → Verify it appears in C:\FTPRoot on the server

After logging in with my domain account, I was able to browse the FTP directory and upload files. The FTP server is now fully functional and integrated with Active Directory.

Verification Commands

Command Prompt — Client 
! Test FTP connection

PowerShell — Server 
# Check IIS is running
Get-Service W3SVC

# Check FTP service is running
Get-Service FTPSVC

# List all IIS sites
Get-IISSite

# List all FTP sites
Get-WebConfiguration -Filter '/system.applicationHost/sites/site' | Where-Object {$_.ftpServer -ne $null}

Key Takeaways

What I Understood

  • IIS can host both websites (HTTP) and FTP sites on the same server
  • FTP with Basic Authentication uses domain credentials but sends passwords in plaintext
  • For production, always use SSL (FTPS) or Windows Authentication for security
  • NTFS permissions control who can read/write files on the server
  • FTP authorization rules control who can connect to the FTP site
  • Firewall must allow TCP port 21 for FTP to work
  • Domain users can authenticate to both IIS and FTP using their AD accounts

IIS vs FTP — When to Use What

Use Case Use IIS (HTTP/HTTPS) Use FTP
Host a website ✅ Yes ❌ No
File sharing/transfers Possible but not ideal ✅ Yes
Large file uploads Limited by web server config ✅ Yes
Public access ✅ Yes Use with caution
Internal file repository ❌ No ✅ Yes
