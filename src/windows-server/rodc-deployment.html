RODC Deployment — Read-Only Domain Controller

What I Learned

I deployed a Read-Only Domain Controller (RODC) for a simulated branch office scenario. The key insight: RODCs provide authentication services without storing writable copies of Active Directory data, making them perfect for locations with less physical security.

Users could still log in and authenticate through the RODC, but password changes and other write operations were referred back to the writable DC at headquarters. This hybrid approach balances convenience with security.

What is RODC?

RODC (Read-Only Domain Controller) is a special type of domain controller that hosts a read-only copy of the Active Directory database. Introduced in Windows Server 2008, it's designed for branch offices or locations where physical security cannot be guaranteed.

RODC vs Writable DC

Feature Writable DC RODC
AD Database Read/Write Read-Only
Password Changes Accepts locally Refers to writable DC
Password Caching All passwords Selective (Password Replication Policy)
Replication Bidirectional One-way inbound only
FSMO Roles Can hold roles Cannot hold any FSMO roles
Use Case Headquarters, data centers Branch offices, remote sites

Why Use RODC?

Security Benefits:
  • If RODC is stolen, attackers get read-only data (no passwords by default)
  • Password Replication Policy controls which passwords are cached
  • Credential theft attack surface is minimized
  • Less risk in locations with poor physical security

RODC Key Features

Feature Description
Read-Only AD Database Local changes not allowed; replication is one-way inbound
Password Replication Policy (PRP) Controls which user passwords are cached on RODC
Credential Caching Caches only approved credentials for faster authentication
Administrator Role Separation Delegate local admin without Domain Admin rights
Read-Only DNS DNS zones are also read-only on RODC
Unidirectional Replication RODC never replicates changes back to writable DCs

My Lab Setup

One writable DC at "headquarters" and one RODC deployed as a branch office domain controller. Both in the threatlens.local domain.

Environment Details

Server Role IP Address Location
DC1 Writable DC 192.168.1.10 Headquarters
RODC1 Read-Only DC 192.168.1.20 Branch Office

Prerequisites

Requirements Before Deploying RODC:
  • Forest functional level: Windows Server 2003 or higher
  • At least one writable Windows Server 2008+ DC in the domain
  • adprep /rodcprep run on schema master (if upgrading from 2003)
  • DNS properly configured

RODC Deployment Steps

Step 1 — Install AD DS Role on RODC1

PowerShell — RODC1 
# Install AD Domain Services role
Install-WindowsFeature AD-Domain-Services -IncludeManagementTools

# Verify installation
Get-WindowsFeature AD-Domain-Services

Step 2 — Configure DNS to Point to Writable DC

PowerShell — RODC1 
# Set DNS to writable DC
Set-DnsClientServerAddress -InterfaceAlias "Ethernet" -ServerAddresses 192.168.1.10

# Verify DNS
Get-DnsClientServerAddress

Step 3 — Promote Server to RODC

PowerShell — RODC1 
Import-Module ADDSDeployment

# Promote to RODC
Install-ADDSDomainController `
    -DomainName "threatlens.local" `
    -Credential (Get-Credential "THREATLENS\Administrator") `
    -ReadOnlyReplica:$true `
    -SiteName "Default-First-Site-Name" `
    -InstallDns:$true `
    -SafeModeAdministratorPassword (ConvertTo-SecureString "P@ssw0rd123" -AsPlainText -Force) `
    -Force:$true

# Server will reboot after promotion
Key Parameter: -ReadOnlyReplica:$true is what makes this an RODC instead of a regular writable DC.

Step 4 — Configure Password Replication Policy

PowerShell — DC1 (Writable DC) 
# Allow a specific user's password to be cached on RODC
Add-ADDomainControllerPasswordReplicationPolicy -Identity "RODC1" `
    -AllowedList "THREATLENS\BranchUser1"

# Deny a user's password from being cached
Add-ADDomainControllerPasswordReplicationPolicy -Identity "RODC1" `
    -DeniedList "THREATLENS\Administrator"

# View current Password Replication Policy
Get-ADDomainControllerPasswordReplicationPolicy -Identity "RODC1"
Important: By default, NO passwords are cached on RODC. You must explicitly allow users via Password Replication Policy. Never allow administrative accounts (Domain Admins, Enterprise Admins) to cache on RODC.

Step 5 — Delegate RODC Administration (Optional)

PowerShell — DC1 
# Create a group for branch office admins
New-ADGroup -Name "Branch Office Admins" `
    -GroupScope Global `
    -GroupCategory Security `
    -Path "OU=Groups,DC=threatlens,DC=local"

# Delegate administration of RODC to this group
# (Done through Active Directory Users and Computers GUI)
# Properties of RODC → Managed By tab

Step 6 — Verify RODC Deployment

PowerShell — RODC1 
# Verify RODC status
Get-ADDomainController -Identity "RODC1"

# Check if it's read-only
Get-ADDomainController -Identity "RODC1" | Select-Object IsReadOnly

# View replication status
repadmin /showrepl

# Check DNS zones (should be read-only)
Get-DnsServerZone

Step 7 — Test Authentication Through RODC

1

Join Client to Domain

Join a client computer to the domain and ensure its DNS points to RODC1 (192.168.1.20)

2

Login as Branch User

Login as a user whose password is allowed in PRP. First login will be slow (authenticates to writable DC), subsequent logins faster (cached credential).

3

Check Cached Credentials

On RODC1, run: Get-ADDomainControllerPasswordReplicationPolicy -Identity RODC1 -RevealedList

After deployment, I verified that users could authenticate through the RODC. Password changes attempted on the RODC client were automatically redirected to the writable DC at headquarters!

Key Takeaways

What I Understood

  • RODCs provide authentication without writable AD database
  • Password Replication Policy controls which passwords are cached
  • -ReadOnlyReplica:$true parameter makes it an RODC
  • Replication is one-way: writable DC → RODC only
  • RODC cannot hold any FSMO roles
  • By default, NO passwords are cached (must be explicitly allowed)
  • Perfect for branch offices with less physical security

When to Use RODC

Use RODC Use Writable DC
Branch offices with poor physical security Headquarters, secure data centers
Remote locations with untrusted admins Locations with trusted IT staff
Places where DC theft is a risk Secure server rooms
DMZ or perimeter networks Internal corporate networks

Password Replication Policy Best Practices

Best Practice Reason
Never cache admin accounts Protects privileged credentials if RODC stolen
Only cache local branch users Minimize credential exposure
Use groups for PRP Easier management than individual users
Review revealed credentials regularly Audit which passwords are cached

Common Mistakes

  • Forgot PRP configuration: Users couldn't authenticate after WAN link failure because no passwords were cached
  • Cached admin passwords: Accidentally allowed Domain Admin in PRP — major security risk
  • Wrong forest functional level: RODC requires minimum Windows Server 2003 forest level
