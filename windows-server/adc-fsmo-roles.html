What I Learned

I promoted a second Windows Server to become an Additional Domain Controller (ADC) in my existing domain, then transferred all five FSMO roles from the original DC to the new one. The biggest insight: redundancy in Active Directory isn't just about backups — it's about having multiple domain controllers sharing the workload and providing failover.

Transferring FSMO roles felt critical. These five roles control specific domain-wide and forest-wide operations. If the server holding them goes down, certain AD operations stop working. Spreading them across DCs or consciously placing them improves resilience.

What is ADC and FSMO?

Additional Domain Controller (ADC)

Additional Domain Controller (ADC) is any domain controller added after the first one. ADCs replicate Active Directory data from existing DCs, providing redundancy, load balancing, and fault tolerance.

Why Add an ADC?
  • Redundancy — If one DC fails, the other continues serving authentication
  • Load balancing — Multiple DCs handle login requests
  • Geographical distribution — Place DCs in different office locations
  • Faster response — Clients authenticate to nearest DC

FSMO Roles (Flexible Single Master Operations)

FSMO roles are five special roles in Active Directory that handle operations that cannot tolerate multiple simultaneous updates. Only one DC can hold each role at a time.

FSMO Role Scope What It Does
Schema Master Forest-wide Controls all schema changes (adding attributes, classes)
Domain Naming Master Forest-wide Controls adding/removing domains from the forest
RID Master Domain-wide Allocates pools of RIDs (Relative IDs) to DCs for creating objects
PDC Emulator Domain-wide Time synchronization, password changes, Group Policy, legacy support
Infrastructure Master Domain-wide Updates cross-domain group memberships

Forest-wide vs Domain-wide Roles

Role Type Roles Unique Per
Forest-wide Schema Master, Domain Naming Master One per entire forest
Domain-wide RID Master, PDC Emulator, Infrastructure Master One per domain
Important: By default, the first DC in a forest holds all five FSMO roles. If that DC fails and you can't recover it, you'll need to seize the roles (forcefully take them) from another DC.

My Lab Setup

Two Windows Server 2019 machines: DC1 (original domain controller) and DC2 (will become ADC). Both in the same domain: threatlens.local.

Environment Details

Server Hostname IP Address Role
DC1 Server-DC1 192.168.1.10 First DC (holds all FSMO roles initially)
DC2 Server-DC2 192.168.1.11 Additional DC (will receive FSMO roles)

Domain Information

Domain Name threatlens.local
Forest Functional Level Windows Server 2016
Domain Functional Level Windows Server 2016

Configuration Steps

Part 1 — Promote DC2 to Additional Domain Controller

Step 1: Install AD DS Role on DC2

PowerShell — DC2 
# Install Active Directory Domain Services role
Install-WindowsFeature AD-Domain-Services -IncludeManagementTools

# Verify installation
Get-WindowsFeature AD-Domain-Services
Alternative: You can also install AD DS through Server Manager → Add Roles and Features → Active Directory Domain Services.

Step 2: Configure DNS Settings on DC2

PowerShell — DC2 
# Set DNS to point to existing DC (DC1)
Set-DnsClientServerAddress -InterfaceAlias "Ethernet" -ServerAddresses 192.168.1.10

# Verify DNS configuration
Get-DnsClientServerAddress -InterfaceAlias "Ethernet"

Step 3: Promote DC2 to Additional Domain Controller

PowerShell — DC2 
# Promote to domain controller
Import-Module ADDSDeployment

Install-ADDSDomainController `
    -DomainName "threatlens.local" `
    -Credential (Get-Credential "THREATLENS\Administrator") `
    -InstallDns:$true `
    -SafeModeAdministratorPassword (ConvertTo-SecureString "P@ssw0rd123" -AsPlainText -Force) `
    -Force:$true

# Server will automatically reboot after promotion
After the reboot, I logged into DC2 and ran Get-ADDomainController — it showed both DC1 and DC2 as domain controllers. Replication was working!

Step 4: Verify Replication

PowerShell — DC2 
# Check replication status
repadmin /showrepl

# Check all domain controllers in domain
Get-ADDomainController -Filter *

# Force replication from DC1
repadmin /syncall /AdeP

Part 2 — Transfer FSMO Roles to DC2

Step 1: Check Current FSMO Role Holders

PowerShell — Any DC 
# View all FSMO role holders
Get-ADDomain | Select-Object InfrastructureMaster, RIDMaster, PDCEmulator
Get-ADForest | Select-Object DomainNamingMaster, SchemaMaster

# Or use netdom
netdom query fsmo

Step 2: Transfer All FSMO Roles to DC2

PowerShell — DC2 
# Transfer all five FSMO roles to DC2
Move-ADDirectoryServerOperationMasterRole -Identity "Server-DC2" `
    -OperationMasterRole SchemaMaster, DomainNamingMaster, PDCEmulator, RIDMaster, InfrastructureMaster `
    -Confirm:$false
Warning: This command transfers ALL roles at once. In production, you might want to transfer them individually to verify each transfer.

Step 3: Transfer FSMO Roles Individually (Alternative Method)

PowerShell — DC2 
# Transfer each role individually
Move-ADDirectoryServerOperationMasterRole -Identity "Server-DC2" -OperationMasterRole SchemaMaster
Move-ADDirectoryServerOperationMasterRole -Identity "Server-DC2" -OperationMasterRole DomainNamingMaster
Move-ADDirectoryServerOperationMasterRole -Identity "Server-DC2" -OperationMasterRole PDCEmulator
Move-ADDirectoryServerOperationMasterRole -Identity "Server-DC2" -OperationMasterRole RIDMaster
Move-ADDirectoryServerOperationMasterRole -Identity "Server-DC2" -OperationMasterRole InfrastructureMaster

Step 4: Verify FSMO Role Transfer

PowerShell — DC2 
# Verify all roles are now on DC2
netdom query fsmo

# Alternative verification
Get-ADDomain | Select-Object InfrastructureMaster, RIDMaster, PDCEmulator
Get-ADForest | Select-Object DomainNamingMaster, SchemaMaster

Using GUI Method (Active Directory Users and Computers)

1

Transfer RID, PDC, and Infrastructure Master

Open Active Directory Users and Computers → Right-click domain → Operations Masters → Transfer each role individually

2

Transfer Schema Master

Register schmmgmt.dll: regsvr32 schmmgmt.dll → Open MMC → Add Snap-in → Active Directory Schema → Right-click root → Change Schema Master

3

Transfer Domain Naming Master

Open Active Directory Domains and Trusts → Right-click root → Operations Master → Change

Seize FSMO Roles (If Original DC Failed)

Emergency Only: Use seizure only if the original DC is permanently offline and cannot be recovered. Never bring the old DC back online after seizing roles.
PowerShell — Seize Roles 
# Seize all FSMO roles (EMERGENCY ONLY)
Move-ADDirectoryServerOperationMasterRole -Identity "Server-DC2" `
    -OperationMasterRole SchemaMaster, DomainNamingMaster, PDCEmulator, RIDMaster, InfrastructureMaster `
    -Force

Key Takeaways

What I Understood

  • ADCs provide redundancy and load balancing for Active Directory
  • DC2 must point its DNS to DC1 before promotion
  • Replication happens automatically after ADC promotion
  • All five FSMO roles default to the first DC in a forest
  • PDC Emulator is the most critical role (time sync, password changes)
  • Transfer = graceful (original DC online), Seize = forceful (original DC dead)
  • netdom query fsmo shows current role holders

FSMO Roles — What Breaks If They're Down

Role Impact If Offline
Schema Master Cannot modify AD schema (rare operation)
Domain Naming Master Cannot add/remove domains from forest
RID Master Cannot create new objects after RID pool exhaustion
PDC Emulator Password changes delayed, time sync issues, GPO problems
Infrastructure Master Cross-domain group memberships not updated

Best Practices for FSMO Roles

Practice Reason
Keep PDC Emulator on most powerful DC Handles most workload (time, passwords, GPO)
Don't put all roles on one DC in production Single point of failure
Document current role holders Easier disaster recovery
Never seize unless DC is permanently dead Creates conflicts if old DC comes back

Common Mistakes

  • Didn't set DNS on DC2: Promotion failed because DC2 couldn't find DC1 — always point DNS to existing DC first
  • Transferred roles before replication completed: Got errors — wait for replication to finish before transferring FSMO roles
  • Used -Force without understanding: Almost seized roles when I should have transferred them — seizure is only for emergencies
